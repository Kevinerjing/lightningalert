# StudyPilot D1 Plan

This folder holds the first-pass Cloudflare D1 schema for a future unified StudyPilot data layer.

## Why this exists

StudyPilot currently has two strong but different modes:

- Local mode:
  - can run Google Classroom sync scripts
  - can write directly to local `data/*.json`
  - can schedule local update tasks
- Cloudflare mode:
  - can host the site and chat endpoint publicly
  - can analyze uploaded files
  - cannot write back into local JSON files on your computer

The D1 plan is the bridge between those two modes. The goal is:

- local sync writes to D1
- cloud chat writes to D1
- both local and cloud pages read the same data

## Files

- `migrations/0001_initial_schema.sql`
  - base schema for tasks, mistakes, uploads, subject cards, upcoming support, science schedule, and classroom sync data
- `migrations/0002_seed_subjects.sql`
  - seeds the core `science`, `math`, and `english` subjects
- `generated/import-current-data.sql`
  - generated by the export script from the current JSON files

## High-level table design

- `subjects`
  - canonical list of subjects
- `uploads`
  - uploaded files from local mode or cloud mode
- `subject_topic_cards`
  - the main study cards shown on Science/Math/English pages
- `subject_topic_sections`
  - flexible section rows for each card
- `upcoming_support_items`
  - "Upcoming lesson support" cards
- `upcoming_support_sections`
  - flexible section rows for upcoming support cards
- `science_schedule`, `science_schedule_units`, `science_schedule_key_dates`
  - current structured science schedule content
- `study_tasks`
  - unified source for `today`, `week`, and `nextWeek`
- `mistakes`
  - unified mistake notebook and review data
- `classroom_sync_runs`, `classroom_courses`, `classroom_topics`, `classroom_items`
  - raw classroom sync snapshots

## How current JSON maps to D1

### Dashboard JSON

- `data/today.json`
  - -> `study_tasks` where `bucket = 'today'`
- `data/week.json`
  - -> `study_tasks` where `bucket = 'week'`
- `data/next-week.json`
  - -> `study_tasks` where `bucket = 'nextWeek'`

### Subject page JSON

- `data/science.json -> topics[]`
  - -> `subject_topic_cards` + `subject_topic_sections`
- `data/math.json -> topics[]`
  - -> `subject_topic_cards` + `subject_topic_sections`
- `data/english.json -> topics[]`
  - -> `subject_topic_cards` + `subject_topic_sections`

### Science-only structures

- `data/science.json -> upcomingSupport[]`
  - -> `upcoming_support_items` + `upcoming_support_sections`
- `data/science.json -> schedule`
  - -> `science_schedule`
- `data/science.json -> schedule.units[]`
  - -> `science_schedule_units`
- `data/science.json -> schedule.keyDates[]`
  - -> `science_schedule_key_dates`

### Mistakes

- `data/mistakes.json -> mistakes[]`
  - -> `mistakes`

### Classroom sync

- `data/classroom-updates.json`
  - -> `classroom_sync_runs`, `classroom_courses`, `classroom_topics`, `classroom_items`

## Migration strategy

Recommended sequence:

1. Keep the current JSON files as the live source while building D1 in parallel.
2. Add local import scripts that copy JSON data into D1.
3. Add cloud writes for uploads/chat updates into D1.
4. Change the pages to read from a small API layer backed by D1.
5. Retire JSON as the primary source only after local and cloud outputs match.

## Export current JSON into SQL

From the repo root:

```powershell
npm.cmd run studypilot:d1:export
```

This generates:

- `studypilot/d1/generated/import-current-data.sql`

After your D1 database is created and bound in `wrangler.toml`, you can apply it with a command like:

```powershell
npx.cmd wrangler d1 execute studypilot --local --file studypilot/d1/generated/import-current-data.sql --config studypilot/wrangler.toml
```

For remote execution later:

```powershell
npx.cmd wrangler d1 execute studypilot --remote --file studypilot/d1/generated/import-current-data.sql --config studypilot/wrangler.toml
```

## Initialize local D1

To build a local D1 database using the current schema plus the current JSON export:

```powershell
npm.cmd run studypilot:d1:init-local
```

This script runs:

1. the JSON export
2. the main schema migration
3. the subject seed migration
4. the generated current-data import

## Auto-sync after local chat updates

The local Node StudyPilot server now does two things after chat uploads write new JSON content:

- first, it tries to mirror the new rows directly into the local Wrangler D1 SQLite database
- if that direct write path is unavailable or fails, it falls back to a full local JSON -> D1 sync

- default: enabled
- disable temporarily:

```env
STUDYPILOT_SYNC_LOCAL_D1=0
```

You can also run the sync manually:

```powershell
npm.cmd run studypilot:d1:sync-local
```

## Create the remote D1 database later

When you are ready to create the real Cloudflare database:

```powershell
npx.cmd wrangler d1 create studypilot --config studypilot/wrangler.toml
```

Wrangler will return:

- `database_name`
- `database_id`
- a suggested `[[d1_databases]]` block

Copy that block into `studypilot/wrangler.toml` by replacing the current commented placeholder.

## Notes

- This schema is intentionally normalized enough for long-term growth but still close to the current page model.
- `subject_topic_sections` and `upcoming_support_sections` are flexible so the existing card sections can evolve without redesigning the schema each time.
- The schema supports dual-source operation through `source_kind` and `source_upload_id`.
